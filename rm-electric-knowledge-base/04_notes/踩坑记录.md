### 2026-07-18 - BMI088 标定 Flash 写入失败（PGSERR）

> **日志系统使用**：本次调试全程依赖日志插桩，ulog 系统的 API 用法和 Ozone Terminal 查看方式见 [[../../02_code_twin/utils/ulog]]。

---

#### 问题现象

2027 年代码中，BMI088 标定过程调用 `BSP_FLASH_Write_Buffer` 写入 Flash 失败，`FLASH->SR = 0xC0`（PGSERR + PGPERR），HAL 返回 `HAL_ERROR`。但调度器启动后的测试线程里同样的写操作却能成功。

---

#### 自己排查，遇到困难

1. **看寄存器、对比 26 年代码**：一步步单步调试，查看外设寄存器标志位。对比 2026 年代码发现，26 年在逐行调试时同样也无法正常写入 Flash。
2. **陷入僵局**：单步调试会改变外设时序（J-Link 停顿影响 Flash 控制器），导致单步出现的错误在全速时不存在；全速时又看不到内部状态。陷入"单步会错、全速看不到"的两难局面。

---

#### AI 破局：日志插桩 + 二分定位

把问题描述给 AI，AI 提出**日志插桩 + 二分法**的思路——这比继续单步调试高效得多。

**AI 读代码**：把 `bsp_flash.c`、`module_bmi088_cali.c`、`stm32f4xx_it.c`、`main.c` 的初始化流程全部喂给 AI，让它理解完整调用链 `main → tx_application_define → Robot_Init → MODULE_Init → Module_BMI088_init → calibrate → BSP_FLASH_Write_Buffer`。

**AI 联网搜同类案例**：AI 搜到了 ST 官方 LAT1210（DMA 中断残留往地址 0 写导致 PGSERR）、硬汉论坛（IWDG 未初始化就 Refresh）、Stack Overflow（栈溢出野指针写 Flash）等多个同类案例，帮我们排除了几个方向。

**AI 生成插桩代码**：AI 自动在 `main.c` 的每个 `MX_xxx_Init` 后面、`robot_init.c` 的 BSP/MODULE/APP Init 之间插入 `FLASH->SR` 检查日志。

**人类全速跑 → AI 分析日志**：发现进入 `Robot_Init` 时 SR 干净，但到 BMI088 调 Flash 时 SR 已脏——问题出在 BSP_Init 内部某个 MX_Init 弄脏了 Flash 标志位。

**解决方案**：在 `BSP_FLASH_Erase_Sector` 和 `BSP_FLASH_Write_Buffer` 入口加 `__HAL_FLASH_CLEAR_FLAG` 清除残留标志。Flash 操作立即成功。

---

#### 从中总结的调试方法论

这个案例本身就是最好的例子——说明为什么嵌入式调试应该优先用 **AI 插桩 + 日志分析 + 全速运行**，而不是传统单步调试。

**逐行调试的致命局限**

| 问题                 | 说明                                                           |
| ------------------ | ------------------------------------------------------------ |
| J-Link 干扰 Flash 时序 | 逐行调试时 J-Link 每步都通过调试接口读 Flash，干扰 Flash 控制器时序，出现"逐行跑失败、全速跑成功" |
| 改变被测系统行为           | 调试器改变了中断响应时序、DMA 传输节奏，问题不可复现                                 |
| 效率极低               | 每次只能看一个断点的状态，要看初始化流程中哪步出问题需要几十次单步                            |

**AI 辅助调试的有效模式**

```
AI 读代码 → 联网搜同类案例 → 生成插桩代码 → 人类烧录跑 → AI 分析日志
```

- **AI 读代码**：把相关源文件喂给 AI，让它理解完整调用链
- **联网搜案例**：AI 搜索同类芯片+同类错误标志的案例，排除方向或找到根因
- **生成插桩代码**：AI 自动在关键位置插入状态检查日志
- **人类烧录跑**：全速运行，不逐行调试
- **AI 分析日志**：把日志贴给 AI，直接定位是哪个步骤出了问题

**根因**：BSP_Init 阶段某个 MX_Init（疑似 ADC DMA 或 SPI DMA 中断）把 FLASH->SR 弄脏了，脏标志残留到标定时。由于日志系统在 main 阶段未初始化，无法精确定位是哪个 Init。

**注意**：
- FLASH->SR 的错误标志是**黏着的**，不清除会持续影响后续操作。操作前清标志是防御性编程
- main 阶段日志系统未初始化时，用 `printf` 直连串口，或在 Ozone 里直接读寄存器值补盲

---

**STM32F407 FLASH_SR 位定义速查**

| Bit | 值 | 标志 | 说明 |
|-----|-----|------|------|
| 0 | 0x01 | EOP | 操作完成 |
| 1 | 0x02 | OPERR | 操作错误 |
| 4 | 0x10 | WRPERR | 写保护错误 |
| 5 | 0x20 | PGAERR | 编程对齐错误 |
| 6 | 0x40 | PGPERR | 编程并行错误 |
| 7 | 0x80 | PGSERR | 编程序列错误 |
| 16 | 0x10000 | BSY | 忙标志 |

> `0xC0` = PGSERR(bit7) + PGPERR(bit6)。注意位映射不要搞混。
